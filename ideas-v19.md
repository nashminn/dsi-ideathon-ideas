# DSI Ideathon 2026 — Ideas (v19) — Agentic, Non-Chatbot, Scheduled/Triggered

Three ideas built around the principle: the agent acts on its own because something happened or because it's time. No user needs to start a conversation.

---

## Idea 61

**Title**
LoanWatch — Automated Loan Portfolio Delinquency Monitor

**The Problem**
Bangladesh's banking sector has one of the highest non-performing loan (NPL) ratios in the region — a persistent problem Bangladesh Bank has been pushing banks to reduce for years. The root cause is not that loans go bad suddenly. It is that early warning signs are there — a missed installment, a partial payment, an irregular payment pattern — and nobody acts on them fast enough because the monitoring is manual.

A loan officer managing 200 active accounts cannot realistically track each account's payment status daily. They learn a loan is in trouble when it is already 60 or 90 days past due and classified. At that point recovery is difficult and credit bureau implications are already in motion. The borrower also often does not receive a timely reminder — they may have simply forgotten, or run into a short-term cash problem that an early nudge would have resolved before it became a default.

Banks have the data to catch this early. Their core banking systems record every repayment in real time. What they do not have is an automated system that watches that data continuously and escalates without anyone asking it to.

**The Product Idea**
A scheduled agent that runs daily against the bank's loan repayment data and manages the entire early warning and escalation ladder autonomously — no human prompting required at any stage.

The escalation logic runs automatically:
- **DPD 1–7**: Agent sends an automated WhatsApp or SMS reminder directly to the borrower. Contextual, not a template — the message references the loan product, the amount due, and the payment channel, generated by LLM from the borrower's account data.
- **DPD 7–30**: Agent escalates to the assigned loan officer with a structured account summary — borrower contact, outstanding amount, payment history, and a suggested call script. Officer has everything they need to make a recovery call without opening the core banking system.
- **DPD 30–60**: Branch manager receives a consolidated alert covering all accounts in this bracket across the branch. No manager has to remember to check — the list comes to them.
- **DPD 90**: Automatic pre-classification alert with CIB reporting preparation. Compliance team is notified before the regulatory deadline, not after.

Beyond reactive escalation, the agent runs a weekly predictive pass: using payment behavior patterns from the portfolio — late-but-paying accounts, accounts showing decreasing payment amounts, accounts with irregular payment timing — it scores which current accounts are statistically likely to go delinquent in the next 30 days. These accounts get flagged for proactive outreach before they miss a payment. The intelligence is in catching the pre-default signal, not just reacting to the miss.

**Who Benefits**
Risk and collections heads at scheduled banks and NBFIs are the buyers. Bangladesh has 61 scheduled banks and 35 NBFIs, all carrying loan portfolios that Bangladesh Bank scrutinises for NPL ratios. A bank that demonstrably reduces its DPD 1–30 bucket through early automated outreach can show Bangladesh Bank a managed portfolio — this is both a regulatory and a financial outcome the bank's board cares about directly.

Pricing per active loan account per month (BDT 20–50/account) aligns the fee with portfolio size and scales naturally. A mid-sized bank with 50,000 active loan accounts at BDT 30/account = BDT 1.5M/month per bank. 20 bank clients = BDT 30M/month recurring.

**Why Now**
Bangladesh Bank has been intensifying NPL reduction pressure since 2023. The scheduled bank NPL ratio is a watched metric at the policy level. Banks are now under formal targets. The tool that helps a bank hit those targets has a regulatory urgency argument behind it, not just a cost-saving one. Core banking API access or data exports are standard at scheduled banks — the integration path exists. LLM-generated contextual outreach messages are now production-quality and cost-effective at the per-account level.

**Feasibility & Scope**
Data input: daily repayment data export from core banking (CSV/API) → DPD calculation per account → escalation rule engine → LLM message generation → WhatsApp/SMS dispatch → weekly predictive risk scoring pass → management reporting dashboard.

POC: one bank, one loan product, DPD 1-7 automated reminder only, 3 weeks to build. Full escalation ladder and predictive scoring is v1 production. Needs: bank data export/API, LLM API for message generation, WhatsApp Business API, ML model trained on historical repayment data (the bank provides this — it exists in their core banking history).

Biggest unknown: bank data access agreements. Addressed by positioning as a bank-deployed tool (DSI hosts, bank controls data access) — same model as any core banking integration vendor.

**Dishonesty Filter**
Not applicable — the agent acts on the bank's behalf using the bank's own data. The bank is the buyer and beneficiary. No self-reported user data involved.

**Tag:** Agentic Platform | Scheduled + Triggered | Recurring SaaS | Fintech

## Idea 62

**Title**
InvoiceIQ — Automated Accounts Payable Processing Agent

**The Problem**
Every organisation that buys goods or services receives invoices. Processing those invoices — reading them, entering data into the accounting system, matching against purchase orders, routing for approval, flagging discrepancies, and tracking payment — is done manually by AP clerks in virtually every Bangladeshi organisation above a certain size.

The manual process is slow, error-prone, and scales linearly with invoice volume. A 200-person organisation receiving 300 invoices per month typically has 1–2 AP staff doing data entry full-time. Errors in invoice matching create duplicate payments, missed payments, and vendor disputes. Invoices pile up at approval bottlenecks because the approver has to log into a system, find the invoice, read it, and approve — so they batch-approve once a week, creating payment delays that damage vendor relationships.

The problem is not unique to large companies. Every hospital, NGO, factory, and bank with a procurement operation runs the same manual process. The volume varies; the pain does not.

**The Product Idea**
A triggered agent that handles the accounts payable workflow from invoice receipt to payment-ready status, acting automatically at each step without human data entry.

The trigger is invoice arrival — via email attachment or WhatsApp document. From there:

1. **Extraction**: LLM with document vision reads the invoice and extracts structured data — vendor name, invoice number, date, line items, quantities, unit prices, VAT, total amount. Handles printed PDFs, scanned images, and varying invoice formats without template configuration per vendor.
2. **PO Matching**: Extracted line items are matched against the organisation's open purchase orders. Full match → routed for approval. Partial match or price discrepancy → flagged with the specific discrepancy highlighted, routed to AP supervisor for review.
3. **Approval Routing**: Based on invoice amount and department, the agent routes to the correct approver via WhatsApp or email with a one-tap approve/reject action. Approver does not need to log into a system — they respond directly. If no response within 48 hours, the agent escalates to the approver's manager.
4. **AP Ledger Update**: On approval, the agent records the payable in the AP ledger and adds the invoice to the next payment batch.
5. **Aging Report**: Every Monday morning, the agent sends the CFO and AP head a payment aging report — invoices by status (pending approval, approved-pending payment, overdue for payment) — without anyone requesting it.

No invoice sits in someone's inbox unprocessed. No payment is missed because an approver forgot to act. The AP team shifts from data entry to exception handling.

**Who Benefits**
Finance and procurement heads at mid-to-large organisations are the buyers: banks, insurance companies, hospitals, NGOs, manufacturing companies, large retailers. Any organisation processing more than 100 invoices per month has a measurable problem that this solves.

Pricing per invoice processed (BDT 50–150/invoice) is transparent and scales with actual usage. An organisation processing 500 invoices/month at BDT 80/invoice = BDT 40,000/month. Accounting firms processing invoices for multiple client companies on outsourced AP pay a volume rate and mark up to their clients — a natural distribution channel.

**Why Now**
Multimodal LLMs can now read invoices in arbitrary formats with production-quality extraction accuracy — this was not true two years ago and is the core technical enabler. Bangladesh's mid-market is growing, transaction volumes are increasing, and finance teams have not scaled proportionally. Digital procurement is expanding (more PO-backed purchasing at organisations that previously ran informal procurement) — creating the PO matching use case that was absent when purchases were verbal.

**Feasibility & Scope**
Invoice intake (email/WhatsApp) → LLM multimodal extraction → PO matching → approval routing → ledger entry → aging report generation.

POC: email invoice intake, extraction for 3 vendor formats, simple PO match, approval via WhatsApp, 3 weeks. Full multi-format extraction and approval routing: 6–8 weeks. Needs: email API or WhatsApp Business API, LLM multimodal API, client's PO data (CSV export or simple integration). No ERP integration required in v1 — the AP ledger is maintained in the product's own database with export capability.

Biggest unknown: invoice format variety. Addressed by the LLM's ability to handle arbitrary layouts — unlike template-based OCR tools, no per-vendor configuration is required.

**Dishonesty Filter**
Not applicable — the organisation is the buyer and the beneficiary. Vendor invoice fraud (inflated amounts) is detected, not enabled, by the PO matching step.

**Tag:** Agentic Platform | Triggered | Recurring SaaS | Finance Automation

## Idea 63

**Title**
GrantPilot — Automated Donor Report Generation for NGOs

**The Problem**
Bangladesh has one of the largest NGO sectors in the world. BRAC alone operates at a scale that rivals government line ministries. Below BRAC: thousands of national and international NGOs — CARE, Save the Children, Concern Worldwide, Plan International, ActionAid, and hundreds of local organisations — run donor-funded programs across health, education, livelihoods, disaster response, and rights.

Every one of these organisations is contractually required to report to their donors quarterly or biannually. Each donor has its own format: USAID requires logframe-based performance reports with FACTS information; FCDO uses Results-Based Management frameworks with indicator tracking tables; UN agencies require narrative reports mapped to results chains; European donors (EU, GIZ, SDC) have their own templates. A single NGO managing five donors writes five different report formats simultaneously.

Program staff — the people doing the actual field work — spend two to four weeks per quarter extracting data from databases, writing narrative sections, computing indicator percentages, tracking expenditure against budget lines, and fitting all of it into each donor's specific format. This is the highest-friction administrative burden in the development sector and it has no automation at all. The reports are written manually in Word, every quarter, forever.

The data exists. Activity records are maintained in program databases. Financial data is in accounting systems. Beneficiary counts come from M&E teams. The problem is not data collection — it is the labour-intensive process of compiling, computing, and writing the report in the right format.

**The Product Idea**
A scheduled agent that runs at the end of each reporting period, compiles program data from the organisation's systems, and drafts the donor report in the required format — ready for program staff to review and approve rather than write from scratch.

The product has two components:

**Data layer**: A lightweight program data entry module where field staff log activities weekly — not a chatbot conversation, just structured form entries: activity type, location, beneficiary count by gender/age, date. This replaces the end-of-quarter data scramble with continuous structured capture. The agent also pulls financial data from the accounting system (CSV export or API).

**Report generation agent**: At the scheduled reporting deadline (quarterly, biannual — configurable per donor), the agent:
1. Computes all required indicators from the accumulated activity data — achievement against target, percentage completion, disaggregation by gender/geography
2. Maps expenditure to donor budget lines
3. Drafts narrative sections — progress against objectives, key achievements, challenges, lessons learned — using LLM to write from the structured data, in the tone and format each donor expects
4. Assembles the complete report in the donor's required template
5. Sends the draft to the program team for review with a structured editing interface — staff correct or expand specific sections rather than writing from scratch
6. Tracks submission deadlines and sends escalating reminders if the review is not completed in time

What takes 3 weeks of staff time per donor becomes 2 days of review and approval. Multiply across five donors: the program team reclaims a month of productive time per quarter.

**Who Benefits**
Program directors and finance managers at NGOs and development sector organisations are the buyers. The procurement process for NGO software tools is well-established — these organisations have IT budgets and vendor management processes.

BRAC, CARE Bangladesh, Save the Children Bangladesh, Plan International, ActionAid, and their equivalents in other countries are large enough to pay institutionally. Below them: the long tail of mid-sized national NGOs managing one or two donor contracts. Pricing per organisation per active donor contract (BDT 20,000–50,000/quarter per contract) aligns cost with reporting burden.

International expansion is the natural growth path. The same reporting problem exists in every country where USAID, FCDO, or UN agencies fund programs — which is every developing country. The donor formats are identical across geographies; only the program data changes. A product built for Bangladesh NGOs works in Kenya, Ghana, Myanmar, and Cambodia without rebuilding.

**Why Now**
LLM-generated narrative text is now of sufficient quality to produce donor report drafts that program staff can edit rather than rewrite — this is the specific capability that unlocks the product. Two years ago, generated text required heavy editing that offset the time savings. Now it does not. At the same time, donor reporting requirements have been increasing in complexity and frequency as accountability frameworks have tightened post-COVID. The reporting burden is growing; the staffing to handle it has not.

**Feasibility & Scope**
Activity data entry module → financial data import → indicator computation → LLM narrative generation per section → template assembly per donor → review workflow → deadline tracking.

POC: one NGO, one donor (USAID format), one quarter's data, 3 weeks to build. Adding additional donor formats is 3–5 days per format once the base system is built. Needs: LLM API, simple form-based data entry module, financial data CSV import. No complex integrations required in v1 — data enters via forms and CSV, not live system connections.

Biggest unknown: LLM narrative quality for sector-specific content. Addressed by: fine-tuning prompt templates against real donor report examples, and designing the review workflow to make editing fast — the agent generates a first draft, not a final submission.

**Dishonesty Filter**
The organisation reports to donors about its own program outcomes. The agent compiles from the organisation's own data. If an NGO inflates beneficiary numbers, that is donor fraud upstream — the agent does not enable it and does not prevent it. The buyer's interest is accurate, on-time reporting, not evasion. The dishonesty risk is not introduced by this product.

**Tag:** Agentic Platform | Scheduled | Recurring SaaS | Development Sector
